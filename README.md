# streamOutApp — Android RTSP-serve APK (CLI Build)

Camera2 + MediaCodec capture, packaged as a foreground service that hosts an
RTSP server on the device. Native side links a self-contained
`libgstreamer_android.so` built from a Cerbero-generated Android arm64 sysroot.

This repository no longer depends on the old shared helper scripts from the
previous project. The supported build story is now:

- Linux runner, host, or Linux Docker container built from
  `platforms/android/docker/Dockerfile`
- Android SDK/NDK and GStreamer sysroot installed **into an in-repo cache**
  at `platforms/android/cache/` by the scripts under
  `platforms/android/scripts/`
- A single `env.sh` at the repository root that defines every version pin
  and every install path
- A small shared env file at `$HOME/shared/platforms/android/env.sh`
  (auto-generated by the scripts) so the unchanged `build_android.sh` can
  find the SDK, NDK, and GStreamer sysroot

The GitHub Actions workflow at `.github/workflows/android.yml` is the canonical
CI mirror of this setup and is the source of truth for version pins.

### 1. Prerequisites

Use a Linux host, a CI runner, or the Docker image defined in
`platforms/android/docker/`. The build scripts assume a Linux shell.

Required toolchain (all pinned in `env.sh` at the repo root):

- JDK 17 (provided by the Docker image; `eclipse-temurin:17-jdk-jammy`)
- Android SDK command-line tools
- Android platform `android-32`
- Android build-tools `36.0.0`
- Android NDK `30.0.14904198`
- Android CMake `3.22.1`
- Cerbero `1.28.4` to build the GStreamer Android arm64 sysroot

`env.sh` also defines the install layout that `install_sdk_ndk.sh` and
`build_gstreamer.sh` produce:

```text
platforms/android/cache/
├── sources/cerbero/                       (CERBERO_HOME)
├── toolchain-36.0.0-30.0.14904198/        (ANDROID_HOME, versioned)
│   └── ndk/30.0.14904198/                 (ANDROID_NDK_HOME)
└── target_sysroot/                        (GSTREAMER_SYSROOT)
```

Bumping any pinned version in `env.sh` creates a fresh `toolchain-*` dir
instead of clobbering the previous one, so rollback is a symlink swap and
CI never has to wipe caches by hand.

Notes:

- This app is `arm64-v8a` only. Gradle, CMake, and `Application.mk` all
  enforce that ABI.
- The Cerbero/GStreamer output must contain `share/gst-android/ndk-build`
  or the native build will fail.
- `.github/workflows/android.yml` is the source of truth for version pins.
- gRPC Java stubs are generated by the `com.google.protobuf` Gradle plugin
  at build time from `app/src/main/grpc/protos/*.proto`; **no manual
  `protoc` invocation is required.**

### 2. Build (Docker image workflow)

All four steps below run inside the Docker builder image, which contains
every host-level tool Cerbero and sdkmanager need. The Android SDK, NDK, and
GStreamer sysroot are **not** in the image — they are produced by steps 2
and 3 into `platforms/android/cache/`, so they survive image rebuilds and
`docker compose down -v`.

#### Step 0 (optional) — reset the in-repo cache

Run this only if a previous step-2/step-3 run left the cache in a bad
state. The Docker image itself is unaffected — we only wipe the in-repo
toolchain / cerbero clone / gstreamer sysroot.

```sh
rm -rf platforms/android/cache
```

#### Step 1 — build the builder image

From the repo root:

```sh
docker compose -f platforms/android/docker/docker-compose.yml build
```

This produces the image `streamoutapp_builder:latest`. The build user inside
the image is `builder` (UID/GID default `1000:1000` — override with
`USER_UID=$(id -u) USER_GID=$(id -g) docker compose ... build` if your host
UID differs, so the bind-mounted repo and `~/.ssh` stay writable/readable).

#### Enter an interactive container

```sh
docker compose -f platforms/android/docker/docker-compose.yml run --rm builder
```

The repo is bind-mounted at `/workspace` (== container `WORKDIR`); the host
`~/.ssh` and `~/.gitconfig` are bind-mounted read-only into the container's
`/home/builder/` so `git` inside the container works with your existing
credentials.

Inside the container:

```sh
. ./env.sh                                       # load pins + cache paths
./platforms/android/scripts/install_sdk_ndk.sh   # Step 2
./platforms/android/scripts/build_gstreamer.sh   # Step 3 (~1h first time)
./build_android.sh debug                          # Step 4
```

#### Recovering from a corrupted Cerbero download cache

Cerbero downloads upstream tarballs into `~/.cache/cerbero-sources/`
(persisted in the `builder-home` named volume, so it survives
`rm -rf platforms/android/cache`). If a download is interrupted, Cerbero
renames the bad file to `*.failed-checksum` and refuses to redownload on
the next run — bootstrap will then abort with a `Checksum for ... is ...
instead of ...` error. Delete both the stale zip and the marker file, then
re-run Step 3:

```sh
rm -f /home/builder/.cache/cerbero-sources/android-ndk-r29-linux.zip \
      /home/builder/.cache/cerbero-sources/android-ndk-r29-linux.zip.failed-checksum
./platforms/android/scripts/build_gstreamer.sh
```

Same trick works for any other file cerbero flags with a checksum
mismatch — just substitute the filename from the error message.


#### One-shot (non-interactive) equivalents

```sh
# Step 2:
docker compose -f platforms/android/docker/docker-compose.yml \
    --profile install run --rm install-sdk-ndk

# Step 3:
docker compose -f platforms/android/docker/docker-compose.yml \
    --profile gstreamer run --rm build-gstreamer

# Step 4:
docker compose -f platforms/android/docker/docker-compose.yml \
    --profile app run --rm build-app
```

#### Step 4 details

`build_android.sh` sources `$HOME/shared/platforms/android/env.sh` (written
for you by the step-2/step-3 scripts), sanity-checks the sysroot, rewrites
`local.properties` from that env so the Android Gradle Plugin uses the same
SDK/NDK, then runs Gradle:

```sh
./build_android.sh               # assembleDebug (default)
./build_android.sh release       # assembleRelease
./build_android.sh debug clean   # ./gradlew clean + wipe app/.cxx + app/src/main/gst/{libs,obj,src,gst-android-build}
./build_android.sh clean         # clean only, no build
```

> **Note — executable bit on `gradlew` / `build_android.sh`.** Both are
> tracked in git as mode `0644` and CI chmods them on every run (see
> `.github/workflows/android.yml`). The `build-app` compose profile does
> the same. If you invoke `./build_android.sh` directly inside a
> freshly-entered `builder` shell and get `Permission denied`, run
> `chmod +x gradlew build_android.sh` once.

Under the hood: AGP invokes CMake at `app/src/main/cpp/CMakeLists.txt`,
which runs `ndk-build -C app/src/main/gst` against `gst/jni/Android.mk`.
The resulting `libgstreamer_android.so` and `libc++_shared.so` are picked
up by AGP from `app/src/main/gst/libs/` (via
`sourceSets.main.jniLibs.srcDirs`) and packaged into the APK.

#### Building on a bare Linux host (no Docker)

The same three scripts work on the host directly — you just have to install
the OS-level packages listed in `platforms/android/docker/Dockerfile`
(autotools, meson/ninja, flex/bison, gettext/gperf, texinfo, nasm/yasm,
python3, JDK 17, …) first. Then:

```sh
. ./env.sh
./platforms/android/scripts/install_sdk_ndk.sh
./platforms/android/scripts/build_gstreamer.sh
./build_android.sh debug
```

Direct `./gradlew assembleDebug` also works, provided
`$HOME/shared/platforms/android/env.sh` has been sourced in the current shell
so CMake can see `GSTREAMER_SYSROOT`.

### 4. Install & run on device

```sh
# Disable CSS so it doesn't compete for camera / RTSP port
adb shell pm disable com.crestron.txrxservice

# Install
adb install -t -g app/build/outputs/apk/debug/app-debug.apk

# Start the foreground service
adb shell am start-foreground-service -n com.crestron.streamout/.StreamOutSvcCtrl

# Or start with a single native gtest filter (all optional)
adb shell am start-foreground-service \
  -n com.crestron.streamout/.StreamOutSvcCtrl \
  -e gtest_filter "GStreamerTest.checkStreamoutColorbar" \
  --ei grpc_port 50051 \
  --es grpc_ip 127.0.0.1 \
  --ez run_gtest true
```

### 5. Camera2 Debug Commands

The Camera2 + MediaCodec path (`cam2_streamout_debug.cpp`) exposes a small set of
runtime debug commands. A command string is dispatched through the gRPC
`StreamoutServerDebug` RPC (`debug_string` field), which forwards it to the native
`cam2_streamer_debug()` handler.

**Format:** `<PREFIX> <COMMAND> [args...]`
- Tokens are separated by spaces or commas.
- The first token is a prefix (e.g. `STREAMOUT_CAM2`) and is skipped by the handler.
- Commands are case-insensitive.

| Command | Args | Description |
|---|---|---|
| `PRINT_ELEMENT_PROPERTY` | `<element_name>` | Look up a named element in the cached running pipeline and log all its properties. |
| `SET_CATEGORY_DEBUG_LEVEL` | `<category_name> <level>` | Set the GStreamer debug threshold for a category (level 0–9). |
| `INSPECT_ELEMENT` | `<factory_name>` | Create a temporary instance of an element factory and log its properties (works without a running pipeline). |
| `SET_MAX_BITRATE` | `<bps \| K/M/G suffix>` | Cap the encoder bitrate; applied on the **next** stream start. `0` = no cap. Accepts plain bps or a `K`/`M`/`G` suffix (decimal, case-insensitive). |
| `SET_RTCP_FEEDBACK` | `<true\|false>` | Advertise `a=rtcp-fb` (nack/pli/fir) in the RTSP SDP; applied on the **next** stream start. Accepts `true/false`, `1/0`, `on/off`, `yes/no`. |
| `SET_PREFER_H265` | `<true\|false>` | Select H.265 (`true`) or H.264 (`false`) encoding; applied on the **next** stream start. Accepts `true/false`, `1/0`, `on/off`, `yes/no`. |
| `SET_STREAM_MODE` | `<0\|1\|2 \| name>` | Select which media streams to serve; applied on the **next** stream start. `0`=`VIDEO_ONLY`, `1`=`AUDIO_ONLY`, `2`=`VIDEO_AND_AUDIO_BOTH`. Accepts the ordinal or the (case-insensitive) name. |
| `SET_AUDIO_SAMPLE_RATE` | `<Hz>` | Set the audio capture sample rate (e.g. `44100`, `48000`); applied on the **next** stream start. Independent of the channel count. Unsupported (non-AAC) rates are ignored. |
| `SET_AUDIO_CHANNELS` | `<1\|2>` | Set the audio channel count (`1`=mono, `2`=stereo); applied on the **next** stream start. Independent of the sample rate. |
| `SET_USE_OPUS` | `<true\|false>` | Select the audio codec: `true`=Opus (raw PCM → GStreamer `opusenc → rtpopuspay`, lower latency), `false`=AAC (MediaCodec). Applied on the **next** stream start. Enabling Opus forces the sample rate to `48000`. Accepts `true/false`, `1/0`, `on/off`, `yes/no`. |
| `SET_AUDIO_LATENCY_DEBUG` | `<true\|false>` | Turn the audio-latency diagnostic logs on/off **live** (no restart): the native `Opus encode latency` probe print, the `[A/V sync]`/`[EMA init]` PTS prints, and the Java `Opus push latency` print. **Default OFF.** Logging only — capture, encode, and the EMA PTS correction always run; when off the probe callbacks early-return at ~zero cost. Accepts `true/false`, `1/0`, `on/off`, `yes/no`. |
| `DUMP_ALL_CONFIG` | _(none)_ | Log a summary of every externally-settable static setting (the **next-start intent**, i.e. what the next `start()` will use) to logcat. Read-only. For the **live** running pipeline use `PRINT_ELEMENT_PROPERTY` instead. |
| `DUMP_PIPELINE_GRAPH` | _(none)_ | On-demand dump of the **live** pipeline: write a Graphviz `.dot` graph to the DOT output dir **and** log the full element/property hierarchy to logcat. Bypasses the periodic-dump timer and its file limit. Logs `no pipeline available` if no client has played yet. Development/debugging aid. |

**`SET_MAX_BITRATE` examples** (all equivalent ways to set common values):
```text
STREAMOUT_CAM2 SET_MAX_BITRATE 2000000   # 2 Mbps
STREAMOUT_CAM2 SET_MAX_BITRATE 20M       # 20 Mbps
STREAMOUT_CAM2 SET_MAX_BITRATE 1500K     # 1.5 Mbps
STREAMOUT_CAM2 SET_MAX_BITRATE 0         # remove the cap
```

**Other examples:**
```text
STREAMOUT_CAM2 INSPECT_ELEMENT amcvidenc-omxqcomvideoencoderhevc
STREAMOUT_CAM2 PRINT_ELEMENT_PROPERTY pay0
STREAMOUT_CAM2 SET_CATEGORY_DEBUG_LEVEL rtspmedia 5
STREAMOUT_CAM2 SET_RTCP_FEEDBACK true
STREAMOUT_CAM2 SET_PREFER_H265 false
STREAMOUT_CAM2 SET_STREAM_MODE 2
STREAMOUT_CAM2 SET_STREAM_MODE VIDEO_ONLY
STREAMOUT_CAM2 SET_AUDIO_SAMPLE_RATE 48000   # native HDMI rate (no resample)
STREAMOUT_CAM2 SET_AUDIO_SAMPLE_RATE 44100   # back to the default
STREAMOUT_CAM2 SET_AUDIO_CHANNELS 2          # stereo
STREAMOUT_CAM2 SET_AUDIO_CHANNELS 1          # mono (rate/channels are independent)
STREAMOUT_CAM2 SET_USE_OPUS true             # Opus audio (forces 48000 Hz)
STREAMOUT_CAM2 SET_USE_OPUS false            # back to AAC
STREAMOUT_CAM2 SET_AUDIO_LATENCY_DEBUG on    # enable latency logs (live, default off)
STREAMOUT_CAM2 SET_AUDIO_LATENCY_DEBUG off   # silence latency logs (zero cost)
STREAMOUT_CAM2 DUMP_ALL_CONFIG               # log all current settings to logcat
STREAMOUT_CAM2 DUMP_PIPELINE_GRAPH           # write .dot graph + log full pipeline (live)
```

Notes:
- These settings are also configurable from `configure.txt` (read on service start):
  `max_bitrate` (integer bps), `rtcp_feedback` (bool), `prefer_h265` (bool), and
  `stream_mode` (integer ordinal 0/1/2).
  Both routes call the matching `Camera2GstStreamer` static setters
  (`setMaxBitRate(int)`, `setRtcpFeedback(boolean)`, `setPreferH265(boolean)`,
  `setStreamMode(int)`).
- `SET_AUDIO_SAMPLE_RATE` / `SET_AUDIO_CHANNELS` are debug-command only (they call
  `setAudioSampleRate(int)` / `setAudioChannelCount(int)`); they are not read from
  `configure.txt`.
- `DUMP_ALL_CONFIG` takes no arguments and is read-only; it calls
  `getConfigSummary()` and logs the result to logcat (the debug dispatcher has no
  return channel). It reports the configured **next-start intent**, not the live
  element state – use `PRINT_ELEMENT_PROPERTY` for the running pipeline.
- These encoder/SDP settings take effect on the next `start()`, not on a stream
  that is already running (the RTSP factory and encoder read them at init time).
